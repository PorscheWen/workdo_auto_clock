# 打卡時間延遲分析

## 📊 打卡時間延遲的可能原因

### 1. **GitHub Actions 啟動延遲** ⚠️ 最常見
- **原因**: Cron job 觸發後，GitHub Actions 需要時間初始化環境
- **典型延遲**: 1-5 分鐘
- **日誌**: 檢查 `程式啟動時間` vs `實際執行時間` 的差異
- **解決方案**:
  - 在 Workflow 中提前觸發（如 17:25 而不是 17:30）
  - 使用 GitHub Actions 的 `jobs.<job_id>.continue-on-error`

### 2. **網路延遲** 🌐
- **原因**: GitHub Actions 的伺服器到 Workdo API 的網路延遲
- **典型延遲**: 0.5-2 秒
- **日誌**: 檢查 `API 耗時`
- **排查**: 
  ```
  ⏱️  API 耗時: X.XX 秒
  ```

### 3. **Workdo API 響應慢**
- **原因**: 服務器負載高或地理位置遠
- **典型延遲**: 1-5 秒
- **日誌**: 檢查 `API 耗時` 是否超過 3 秒
- **解決方案**: 
  - 檢查 Workdo 服務狀態
  - 在低峰期測試
  - 考慮使用 CDN 或代理

### 4. **登入延遲**
- **原因**: 每次打卡都需要登入，登入需要額外時間
- **典型延遲**: 1-3 秒
- **日誌**: 檢查 `登入耗時`
- **優化**: 考慮複用 Session 以減少登入次數

### 5. **系統時間不同步** 🕐
- **原因**: 服務器時間與本地時間不同步
- **症狀**: 打卡時間與預期時間偏差大
- **日誌**:
  ```
  目前時間 (UTC):      2026-06-25 10:30:45 UTC
  目前時間 (台灣):     2026-06-25 18:30:45 CST
  ```
- **解決方案**: 檢查系統時間是否正確

### 6. **多步驟工作流延遲**
- **原因**: 如果 Workflow 中有多個步驟，前面步驟會造成延遲
- **日誌**: 檢查 `程式啟動時間` 與 Workflow 開始時間的差異
- **優化**: 簡化工作流步驟

## 🔍 診斷打卡延遲

### 檢查日誌中的時間信息

```
===============================================================
⏱️  時間診斷信息
===============================================================
程式啟動時間 (UTC): 2026-06-25 10:12:45 UTC
程式啟動時間 (台灣): 2026-06-25 18:12:45 CST
目前時間 (UTC):      2026-06-25 10:30:45 UTC
目前時間 (台灣):     2026-06-25 18:30:45 CST
程式已執行:         18.50 秒
===============================================================
```

### 打卡過程中的時間記錄

```
⏰ 執行下班打卡...
   系統時間 (打卡前): 18:12:30
   Workdo 記錄時間: 2026-06-25 18:12:35
   系統時間 (打卡後): 18:12:36
   ⏱️  API 耗時: 2.30 秒
   ⏱️  打卡全程耗時: 6.12 秒
```

### 時間延遲判斷

| 項目 | 延遲時間 | 判斷 |
|------|---------|------|
| API 耗時 | < 1 秒 | ✅ 正常 |
| API 耗時 | 1-3 秒 | ⚠️ 輕微延遲 |
| API 耗時 | > 3 秒 | 🔴 嚴重延遲 |
| 打卡全程 | < 5 秒 | ✅ 正常 |
| 打卡全程 | 5-10 秒 | ⚠️ 輕微延遲 |
| 打卡全程 | > 10 秒 | 🔴 嚴重延遲 |

## 📋 延遲排查清單

### 如果打卡失敗

- [ ] 檢查 `程式啟動時間` 是否超過預期（GitHub Actions 延遲）
- [ ] 檢查 `API 耗時` 是否超過 3 秒（網路或服務器延遲）
- [ ] 檢查 `登入耗時` 是否超過 2 秒（服務器延遲）
- [ ] 檢查系統時間是否正確（UTC vs 台灣時間）
- [ ] 查看打卡結果的 `Workdo 記錄時間` 是否超過 18:15

### 如果打卡時間晚於截止

```
下班打卡時間晚於截止時間 
⚠️ 下班打卡時間 18:16:30 晚於截止時間 18:15:00
```

**原因分析**:
1. GitHub Actions 啟動太晚（應該提前到 17:25-17:30）
2. API 響應延遲（> 3 秒）
3. 安全截止時間設置太寬松（目前是提前 3 分鐘，可能需要提前 5 分鐘）

**解決方案**:
- 調整安全截止時間: `CLOCK_OUT_SAFE_CUTOFF_MINUTE = 10` (提前 5 分鐘)
- 調整 Workflow 的 cron 時間提前執行
- 檢查 Workdo 服務是否正常

## 🔧 調優建議

### 短期解決方案

1. **提前執行時間**
   ```yaml
   # .github/workflows/auto-clock.yml
   - cron: '25 9,10 * * *'  # 提前到 17:25 而不是 17:30
   ```

2. **增加安全緩衝時間**
   ```python
   CLOCK_OUT_SAFE_CUTOFF_MINUTE = 10  # 提前 5 分鐘而不是 3 分鐘
   ```

3. **添加重試機制**
   ```python
   # 在 punch() 中添加重試邏輯
   for retry in range(3):
       if self.punch(punch_type):
           break
   ```

### 長期解決方案

1. **使用更可靠的服務**
   - 評估其他打卡系統
   - 考慮使用 API 網關加快響應

2. **優化網路連接**
   - 使用 VPN 或代理以減少延遲
   - 監控 GitHub Actions 的網路性能

3. **增強監控**
   - 定期檢查日誌中的延遲指標
   - 設置告警機制

## 📈 監控指標

建議監控以下指標:

- **登入耗時**: 應在 1-2 秒內
- **API 耗時**: 應在 0.5-2 秒內  
- **打卡全程耗時**: 應在 3-8 秒內
- **程式啟動延遲**: 程式啟動時間 vs Workflow 開始時間

## 🆘 常見問題

### Q: 為什麼安全截止時間設為 18:12？
A: 確保即使 API 有延遲，打卡時間仍在 18:15 內。提前 3 分鐘可容納大多數延遲情況。

### Q: 如何判斷是 GitHub Actions 延遲還是 API 延遲？
A: 檢查日誌中的 `程式已執行` 時間。如果大於預期，說明是 Actions 延遲。

### Q: 為什麼有時打卡時間與系統時間相差幾秒？
A: Workdo 服務器收到請求到記錄打卡時間之間有延遲，這是正常的。
